# concurrent-request-handling Specification

## Purpose

IR 송신 요청을 비블로킹 큐 기반으로 처리하여, 송신이 진행 중인 동안에도 서버가 새로운 HTTP 요청을 받아 응답할 수 있도록 하고, 단일 `IrSender` 인스턴스에 대한 핀 전환·타이밍 경합을 방지한다.

## Requirements

### Requirement: 송신 요청 큐잉과 즉시 응답
모든 IR 송신 요청(`POST /api/ir/{id}/send`, `POST /api/ir/all/send`)은 실제 IR 송신 완료를 기다리지 않고, 요청을 IR 작업 큐에 적재한 뒤 클라이언트에 즉시 정상(2xx) 응답을 반환해야 한다(SHALL). 다른 송신이 진행 중이더라도 새 요청은 거절되지 않고 큐에 누적되어야 하며(SHALL), 큐에 적재된 요청은 유실 없이 순차적으로 처리되어야 한다(SHALL).

#### Scenario: 송신 요청 적재 후 즉시 응답
- **WHEN** 클라이언트가 단일 또는 전체 송신을 요청한다
- **THEN** 시스템은 해당 요청을 IR 작업 큐에 적재한다
- **AND** IR 송신 완료를 기다리지 않고 즉시 정상(2xx, "queued") 응답을 반환한다

#### Scenario: 송신 진행 중 새 요청 누적
- **WHEN** 이전 송신 작업이 아직 처리 중인 상태에서 새 송신 요청이 들어온다
- **THEN** 시스템은 새 요청을 거절하지 않고 큐에 누적한다
- **AND** 진행 중이던 작업이 끝난 뒤 누적된 요청을 순서대로 처리한다

#### Scenario: 연속/동시 요청 무유실 처리
- **WHEN** 서로 다른 여러 송신 요청이 짧은 시간 안에 연속으로 들어온다(큐 용량 이내)
- **THEN** 모든 요청이 큐에 적재되어 하나도 누락되지 않고 각각 송신된다
- **AND** 단, 연속 중복 디듀프 대상(직전 작업과 동일)인 요청은 예외로 한다

### Requirement: 연속 중복 요청 디듀프
시스템은 직전에 적재된 작업(큐가 비어 있으면 현재 송신 중인 작업)과 `target`, `protocol`, `address`, `command`가 모두 동일한 — 즉 완전히 같은 IR 신호의 — 송신 요청이 연속해서 들어오면, 중복 작업을 큐에 새로 추가하지 않아야 한다(SHALL). 사이에 다른 신호 요청이 끼어든 비연속 동일 요청은 디듀프하지 않고 각각 적재되어야 한다(SHALL). 네 필드 중 하나라도 다르면(예: `command`가 같아도 `protocol`/`address`가 다르면) 서로 다른 신호로 보고 각각 적재되어야 한다(SHALL). 디듀프로 큐에 추가되지 않은 요청도 클라이언트에는 정상(2xx) 응답을 반환해야 한다(SHALL).

#### Scenario: 연속 동일 신호 병합
- **WHEN** 직전 적재(또는 현재 송신 중) 작업과 `target`·`protocol`·`address`·`command`가 모두 동일한 요청이 바로 이어서 들어온다
- **THEN** 시스템은 중복 작업을 큐에 추가하지 않는다
- **AND** 해당 요청에도 정상(2xx) 응답을 반환한다

#### Scenario: 사이에 다른 요청이 끼면 병합하지 않음
- **WHEN** 동일한 신호 요청 사이에 다른 `target`/`protocol`/`address`/`command` 요청이 들어온다
- **THEN** 나중에 들어온 동일 요청은 디듀프되지 않고 별도 작업으로 큐에 적재된다

#### Scenario: 일부 필드만 같으면 병합하지 않음
- **WHEN** 직전 작업과 `command`는 같지만 `protocol` 또는 `address`가 다른 요청이 연속해서 들어온다
- **THEN** 서로 다른 신호로 간주되어 디듀프되지 않고 별도 작업으로 큐에 적재된다

#### Scenario: 디듀프되어도 송신은 보장
- **WHEN** 연속 동일 요청이 디듀프되어 큐에 1건만 남는다
- **THEN** 해당 `target`·`command`에 대한 IR 송신은 최소 1회 정상 수행된다

### Requirement: 비블로킹 큐 소비 상태머신
시스템은 IR 작업 큐를 블로킹 `delay()` 누적이 아닌 `millis()` 기반 상태머신으로 소비해야 한다(SHALL). 한 번의 `loop()` 반복에서는 송신 시점이 도래한 단위 작업만 처리하고 즉시 반환하여, 큐를 소비하는 동안에도 서버가 새 클라이언트 요청을 받을 수 있어야 한다(SHALL). 전체 송신 작업은 모듈 간 정의된 간격을 두고 각 모듈을 1회씩 순차 송신해야 한다(SHALL).

#### Scenario: 큐 소비 중 다른 요청 처리
- **WHEN** 큐에 작업이 남아 송신이 진행되는 동안 모듈 간 간격 대기 구간이다
- **THEN** `loop()`는 해당 구간 동안 새로운 HTTP 클라이언트 요청을 받아 응답할 수 있다

#### Scenario: 전체 송신의 순차 진행
- **WHEN** 전체 송신 작업이 큐에서 꺼내져 처리된다
- **THEN** 각 모듈은 정의된 모듈 간 간격 이상이 경과한 시점에 순서대로 1회씩 송신된다
- **AND** 마지막 모듈까지 송신되면 해당 작업이 완료로 처리되고 다음 큐 작업으로 진행한다

#### Scenario: 작업 완료 후 다음 작업 자동 진행
- **WHEN** 현재 작업의 송신이 모두 끝나고 큐에 다음 작업이 남아 있다
- **THEN** 시스템은 추가 외부 트리거 없이 다음 작업을 꺼내 송신을 이어간다

### Requirement: 단일 IR 송신 경로 일원화
실제 `IrSender` 접근(`setSendPin()` 동적 핀 전환 및 송신)은 큐 소비 상태머신 한 곳에서만 수행되어야 한다(SHALL). HTTP 요청 핸들러는 송신을 직접 수행하지 않고 큐 적재만 담당하여, 단일 `IrSender` 인스턴스에 대한 핀 전환·타이밍 경합이 발생하지 않도록 보장해야 한다(SHALL).

#### Scenario: 핸들러는 송신하지 않음
- **WHEN** HTTP 송신 요청이 처리된다
- **THEN** 요청 핸들러는 큐 적재만 수행하고 직접 `IrSender`를 호출하지 않는다

#### Scenario: 송신 경로 단일화
- **WHEN** 여러 작업이 큐에 쌓여 처리된다
- **THEN** 동시에 둘 이상의 작업이 `IrSender`/핀을 전환·송신하지 않고 항상 한 번에 하나씩만 송신된다

### Requirement: 큐 용량 한계 처리
IR 작업 큐는 고정 크기 링 버퍼로 구현되며, 정상 사용 시 버스트를 수용할 수 있도록 충분한 용량을 가져야 한다(SHALL). 극단적으로 큐가 가득 찬 상태에서 새 요청이 들어오면, 시스템은 임의로 무한 대기에 빠지지 않고 큐 포화를 알리는 명확한 응답을 반환해야 한다(SHALL).

#### Scenario: 큐 포화 시 명확한 응답
- **WHEN** 큐가 가득 찬 상태에서 새 송신 요청이 들어온다
- **THEN** 시스템은 블로킹 없이 큐 포화를 나타내는 에러 응답(예: 503)을 반환한다
- **AND** 기존 큐에 적재된 작업의 처리는 영향을 받지 않는다

### Requirement: HTTP 읽기 타임아웃 상한
HTTP 요청 헤더 및 본문(body) 읽기는 응답이 느리거나 끊긴 클라이언트가 서버를 장시간 점유하지 못하도록, 현행(헤더 3000ms, body 2000ms)보다 짧은 타임아웃 상한을 사용해야 한다(SHALL). 타임아웃 도달 시 시스템은 해당 연결을 정리하고 다음 클라이언트 처리로 진행해야 한다(SHALL).

#### Scenario: 느린 헤더 전송에 대한 타임아웃
- **WHEN** 클라이언트가 헤더를 단축된 타임아웃 시간 내에 완전히 전송하지 못한다
- **THEN** 시스템은 타임아웃 후 연결을 정리하고 다음 요청 처리로 넘어간다
- **AND** 해당 클라이언트가 다른 요청 처리를 단축된 타임아웃 시간 이상으로 묶지 않는다

#### Scenario: 정상 속도 요청은 영향 없음
- **WHEN** 클라이언트가 헤더와 본문을 단축된 타임아웃 내에 정상적으로 전송한다
- **THEN** 시스템은 요청을 정상 파싱하여 처리한다

### Requirement: 큐 및 송신 진행 상태 노출
상태 조회 API(`GET /api/status`)는 IR 작업 큐의 대기 작업 수와 현재 처리 중인 작업의 진행 정도(예: 전체 송신의 송신 완료 모듈 수)를 조회할 수 있도록 상태 정보를 응답에 포함해야 한다(SHALL).

#### Scenario: 처리 중 상태 조회
- **WHEN** 큐에 작업이 있거나 송신이 진행 중인 동안 `GET /api/status`를 호출한다
- **THEN** 응답에 대기 중인 작업 수와 현재 작업의 진행 정도(예: 송신된 모듈 수)가 포함된다

#### Scenario: 유휴 상태 조회
- **WHEN** 큐가 비어 있고 진행 중인 송신이 없는 상태에서 `GET /api/status`를 호출한다
- **THEN** 응답에 대기 작업이 없고 처리 중이 아님이 표시된다
